### Answer ID: 67847715
<p>The same problem I faced while working on localhost and enabled the csrf token to true in the config file. I tried all solutions posted on StackOverflow and finally solved it by myself.</p>
<p>I made the changes in Session Variables inside the config.php file and replaced the following code</p>
<pre><code>$config['sess_driver'] = 'files';
$config['sess_cookie_name'] = 'ci_session';
$config['sess_expiration'] = 7200;
$config['sess_save_path'] = NULL;
$config['sess_match_ip'] = FALSE;
$config['sess_time_to_update'] = 300;
$config['sess_regenerate_destroy'] = FALSE;
</code></pre>
<p>with</p>
<pre><code>$config['sess_cookie_name']     = 'ci_session';
$config['sess_expiration']      = 7200;
$config['sess_expire_on_close'] = FALSE;
$config['sess_encrypt_cookie']  = FALSE;
$config['sess_use_database']    = TRUE;
$config['sess_table_name']      = 'core_sessions';
$config['sess_match_ip']        = FALSE;
$config['sess_match_useragent'] = TRUE;
$config['sess_time_to_update']  = 300;
</code></pre>
<p>And after that, you will get an error related to the database Table '.core_sessions' doesn't exist because this time we are storing session in the database so you must create a table that will store the session values with the following SQL Query in your database.</p>
<pre><code>CREATE TABLE IF NOT EXISTS `core_sessions` ( `id` varchar(128) NOT NULL, `ip_address` varchar(45) NOT NULL, `timestamp` int(10) UNSIGNED NOT NULL DEFAULT 0, `data` blob NOT NULL ) ENGINE=InnoDB DEFAULT CHARSET=utf8
</code></pre>
<p>If you using the form_open helper function then you don't have to add anything extra like the csrf token and its hash value whereas the HTML form required you to add a hidden field with the csrf token name and its value.</p>
<pre><code>&lt;input type=&quot;hidden&quot; 
               name=&quot;&lt;?php echo $this-&gt;security-&gt;get_csrf_token_name();?&gt;&quot; 
               value=&quot;&lt;?php echo $this-&gt;security-&gt;get_csrf_hash();?&gt;&quot;&gt;
</code></pre>
<p>I hope this will work for you as well as for upcoming visitors. Thanks</p>

<p>I did had the same problem. There wasn't any issue with the configuration or any code related bug.</p>

<p>(In my case)
The problem was the form's URL was like <code>http://localhost/project/form</code>
but the form was submitted to <code>http://[::1]/project/form/create</code></p>

<p>Problem root was the domain name where the CSRF token was generated and the domain where they were checked. Simply changing the form's URL to <a href="http://[::1]/project/form" rel="nofollow noreferrer">http://[::1]/project/form</a> resolved the problem with my project.</p>

<p>It was just a minor workaround, this issue never occurred in the actual production domain</p>

### Answer ID: 58259132

<p>I ran into this issue an although some of the answers above were helpful there are a few other things to consider when dealing with csrf.</p>

<p>Starting from the top, and to make this as simple as possible. </p>
