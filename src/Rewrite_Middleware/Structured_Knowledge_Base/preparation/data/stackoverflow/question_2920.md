# MySQL ignoring index for count query on INNODB table in MySQL 5.7
[Link to question](https://stackoverflow.com/questions/58508716/mysql-ignoring-index-for-count-query-on-innodb-table-in-mysql-5-7)
**Creation Date:** 1571762021
**Score:** 1
**Tags:** mysql, hibernate, indexing, mysql-5.7
## Question Body
<p>we've got one old database running MySQL 5.1. We now want to migrate it to MySQL 5.7 but some queries that worked okay are suddenly very very slow (by a factor of 60 or more slower).</p>

<p>The INNODB table in question (EVENT) has amongst other columns a COMPANY_ID (foreign key to a COMPANY table) and EVENT_DATETIME of type DATETIME. There is an index on COMPANY_ID, EVENT_DATETIME and for testing I've added one EVENT_DATETIME, COMPANY_ID.
Currently basically all EVENTs have the COMPANY_ID 1 but this will change.</p>

<p>We have a count query to query the number of events in the last year:</p>

<pre><code>select count(distinct this_.EVENT_ID) as y0_ from EVENT this_
       where this_.EVENT_DATETIME&gt;='2018-10-22 00:00:00'
         and this_.EVENT_DATETIME&lt;='2019-11-21 00:00:00'
         and this_.COMPANY_ID = 1;
</code></pre>

<p>The result is around 1,000,000 rows and used to take about 1.5 seconds now it takes up to 100 seconds. While the query on MySQL 5.1 uses an index on COMPANY_ID and EVENT_DATETIME the index is ignored on MySQL 5.7. It seems if MySQL sees that it has to parse too many rows it gives up on an index even if it would help. If I reduce the window to e.g. 10 month MySQL 5.7 uses the index again.</p>

<p>So on MySQL 5.1 the index COMPANY_ID,EVENT_DATETIME is used
on MySQL it only uses a foreign key index for COMPANY_ID.</p>

<p>If I run the query without the where on the COMPANY_ID </p>

<pre><code>select count(distinct this_.EVENT_ID) as y0_ from EVENT this_ 
       where this_.EVENT_DATETIME&gt;='2018-10-22 00:00:00'
         and this_.EVENT_DATETIME&lt;='2019-11-21 00:00:00';
</code></pre>

<p>the query is a lot faster.</p>

<p>Is there any way to force MySQL 5.7 to use a certain index?</p>

<p>If I rewrite the query to this:</p>

<pre><code>select count(distinct this_.EVENT_ID) as y0_ from EVENT this_
     where this_.EVENT_DATETIME&gt;='2018-10-22 00:00:00'
       and this_.EVENT_DATETIME&lt;='2019-11-21 00:00:00'
     GROUP BY COMPANY_ID HAVING COMPANY_ID = 1;
</code></pre>

<p>it is back to about 1 to 1.5 seconds. The problem is we might have more than one of these queries and the queries are generated by Hibernate Criterias which do not support HAVING so my workaround won't work in real life.</p>

## Answers
### Answer ID: 58529039
<p>The query planner may be taking wrong decisions based on the available statistics. You can try to run <code>ANALYZE</code> (<a href="https://dev.mysql.com/doc/refman/5.6/en/analyze-table.html" rel="nofollow noreferrer">https://dev.mysql.com/doc/refman/5.6/en/analyze-table.html</a>) to rebuild the stats and provide better numbers to the planner. Just notice that ANALYZE will block the table while it runs (it is fast). </p>

<p><strong>UPDATE</strong></p>

<p>Reading the MySQL documentation, I found this paragraph:</p>

<blockquote>
  <p>Prior to MySQL 5.7.18, InnoDB processes <code>SELECT COUNT(*)</code> statements by scanning the clustered index. As of MySQL 5.7.18, InnoDB processes <code>SELECT COUNT(*)</code> statements by traversing the smallest available secondary index unless an index or optimizer hint directs the optimizer to use a different index. If a secondary index is not present, the clustered index is scanned.</p>
</blockquote>

<p>Ref: <a href="https://dev.mysql.com/doc/refman/5.7/en/group-by-functions.html#function_count" rel="nofollow noreferrer">https://dev.mysql.com/doc/refman/5.7/en/group-by-functions.html#function_count</a></p>

<p>It meas that the count behavior changed exactly on the version you are using. It may explain the difference. </p>

### Answer ID: 59402950
<p>The optimal index is</p>

<pre><code>INDEX(COMPANY_ID, EVENT_DATETIME, EVENT_ID)  -- in this order
</code></pre>

<p>I appears that your date range is 1 year + one day + 1 second.  Was that deliberate?</p>

<p>If <code>EVENT_ID</code> is the <code>PRIMARY KEY</code> (Please provide <code>SHOW CREATE TABLE</code>), then <code>COUNT(DISTINCT EVENT_ID)</code> could be simply <code>COUNT(*)</code>.</p>
