## Question Body
<p>I am currently working on a internal legacy project and to get more familiar with the backend logic and database design, I have decided to rewrite all queries from the old fashion SQL-86 standard into a more modern one.
But I am stuck on one big view and one participial join (this example is very simplified):</p>
<pre><code>select t4.id
from
    t1,
    t2,
    t3,
    t4
where 
    t2.t1id = t1.id(+)
    and t2.tmpid = t3.tmpid(+)
    and t4.id = t3.t4id(+)
group by t4.id 
having
    count (t2.t1id) = count(t3.tmpid);
</code></pre>
<p>which is based on the following (example) table structure:</p>
<pre><code>create table t1 (
    id number generated by default on null as identity primary key,
    title varchar(100)
);

create table t2 (
    t1id number not null,
    tmpid number not null
);

create table t3 (
    tmpid number not null,
    t4id number not null
);

create table t4 (
    id number generated by default on null as identity primary key,
    title varchar(100)
);

insert into t1(id, title) values(1, 'Order1');
insert into t1(id, title) values(2, 'Order2');

insert into t4(id, title) values(1, 'Attribute 1');
insert into t4(id, title) values(2, 'Attribute 2');

insert into t2(t1id, tmpid) values(1, 1);
insert into t2(t1id, tmpid) values(1, 2);
insert into t2(t1id, tmpid) values(2, 1);
insert into t2(t1id, tmpid) values(2, 2);

insert into t3(tmpid, t4id) values(1, 1);
insert into t3(tmpid, t4id) values(2, 2);
insert into t3(tmpid, t4id) values(2, 1);
</code></pre>
<p>The query in question results in the following result when removing the &quot;group by&quot; and &quot;having&quot; statement:</p>
<pre><code>select *
from
    t1,
    t2,
    t3,
    t4
where 
    t2.t1id = t1.id(+)
    and t2.tmpid = t3.tmpid(+)
    and t4.id = t3.t4id(+)
order by t1.id
</code></pre>
<p><a href="https://i.sstatic.net/vSboa.png" rel="nofollow noreferrer"><img src="https://i.sstatic.net/vSboa.png" alt="enter image description here" /></a></p>
<p>First of all I thought the <code>and t4.id = t3.t4id(+)</code> was just a &quot;wrong&quot; (well it works for over a decade now so it can't really be wrong) specified old fashioned join and just replaced it with a <code>and t3.t4id = t4.id(+)</code> so it would be a &quot;left join&quot; on ANSI SQL, but then the rows with a &quot;null&quot; values just disappears and the <code>having count(t2.t1id) = count(t3.tmpid)</code>  results to true on every row.</p>
<p>So my question is to what resolves the <code>and t4.id = t3.t4id(+)</code> in a ANSI SQL and why? Out of desperation I have tried it with every join and I can't reproduce the null values which are mandatory for the &quot;having&quot;.</p>
<p>Or is there is there another way to implement this query in a modern way?</p>

## Answers
### Answer ID: 75574550
<p>There are two tables joined with (+): <code>t1</code> and <code>t3</code>. So what you have is</p>
<pre><code>from t2
cross join t4
left join t1 on t1.id = t2.t1id
left join t3 on t3.tmpid = t2.tmpid
             and t3.t4id = t4.id
</code></pre>
<p>In your create table statements <code>t2.t1id</code> is a not null column. With a proper foreign key constraint, <code>t2.t1id</code> would always have a parent row match in <code>t1.id</code> and the <code>t1</code> join would just be an inner join really.</p>
<p>If you are working with Oracle's SQL Developer, you can have the program re-write the queries from old Oracle proprietary joins to ANSI joins by the way. See thatjeffsmith's answer here: <a href="https://stackoverflow.com/a/69284420/2270762">https://stackoverflow.com/a/69284420/2270762</a>.</p>
