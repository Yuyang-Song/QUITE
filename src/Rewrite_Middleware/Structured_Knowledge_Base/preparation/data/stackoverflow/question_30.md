<p>Followup, after the tkprof file was posted:</p>

<p>I see you managed to get the tkprof output, but unfortunately you didn't disconnect your session before creating the tkprof file. Now, the cursor was left open and it failed to write STAT# lines to your trace file. This is why you don't have a plan / row source operation in your tkprof file. It would be nice if you can repeat the process, if the suggestion below turns out to be rubbish.</p>

<p>A little speculation from my side: I think the DISTINCT is almost a no-op because you are selecting so many columns. If this is true, then your predicate "WHERE STATUS_ID = 'ORDER_COMPLETED'" is very selective and you will benefit from having an index on this column. After you create the index, make sure you analyze it properly, maybe even with a histogram on if data values are skewed. The end result will be a different plan for this query, starting with an INDEX RANGE SCAN, followed by a TABLE ACCESS BY ROWID, leading to a very fast query.</p>

<p>After you have created an index, you can have the table re-analyzed, including histograms using this statement:</p>

<p>exec dbms_stats.gather_table_stats([owner],[table_name],cascade=>true,method_opt=>'FOR ALL INDEXED COLUMNS SIZE ')</p>

<p>Regards,
Rob.</p>

### Answer ID: 1054852
<p>When troubleshooting applications where I don't have control of the SQL I find that the dbms_sqltune package saves a lot of time.  See <a href="http://download.oracle.com/docs/cd/B28359_01/appdev.111/b28419/d_sqltun.htm" rel="nofollow noreferrer">http://download.oracle.com/docs/cd/B28359_01/appdev.111/b28419/d_sqltun.htm</a> , and yes, unfortunately you should be licensed to use it.</p>

<p>There are procedures in this package to run a tuning analysis against a specific sql_id in the shared pool or the AWR repository.  The analysis will contain indexing recommendations if there are any improvements to be made with additional indexes.  More importantly, the analyzer might discover an improved access path that it can implement with what Oracle calls a SQL Profile - this is a set of hints that will be stored and used whenever this sql_id is executed.  This happens without requiring the hints to be coded in the SQL statement, and there's also an option to do what you can think of as fuzzy matching if your application generates literal values in the statement instead of bind variables.</p>

<p>Of course this tool shouldn't be a substitute for understanding the application and it's data structures, but reading through the output that details the execution path of a better plan (if found) can be educational.</p>

### Answer ID: 1053360
<p>Oracle is accessing the whole table each time you run the query ( TABLE ACCESS (FULL) ).
Creating an INDEX on the STATUS_ID and ORDER_TYPE_ID columns</p>

<pre><code>CREATE INDEX ERP.ORDER_HEADER_I1 ON ERP.ORDER_HEADER ( STATUS_ID, ORDER_TYPE_ID );
</code></pre>

<p>will help a lot, especially if there are several different values of STATUS_ID and ORDER_TYPE_ID in the ORDER_HEADER table.</p>

### Answer ID: 1052779
<p>Is ORDER_ID declared as the PK using a PRIMARY KEY constraint?  Because if it is I would expect the optimiser to recognise that the DISTINCT is superfluous in this query and optimise it out.  Without the constraint, it won't know it is superfluous and so will expend unnecessary and considerable effort in "de-duping" the results.</p>

### Answer ID: 1051683
<p>Since you're ordering results according to <code>order_date</code> it is important that you have a descending index on that field.
Also tell oracle that you wish to use this index. Place the <code>order_date</code> field first in the query and use a hint like</p>

<pre><code>SELECT /*+ index(HEADERS IDX_ORDER_DATE_DESC) */ ... 
FROM ERP.ORDER_HEADER HEADERS
WHERE ...
ORDER BY ORDER_DATE DESC
</code></pre>

<p>It is not so much about having indices, but rather about telling oracle to use them. Oracle is very picky about indices. You get the best results when you choose indices according to your most important queries. If in doubt, <a href="http://www.orafaq.com/wiki/SQL_Trace" rel="nofollow noreferrer">trace a query</a>. This way you can see in what part of the query oracle spends the most time and whether your indices are actually picked up or not. Tracing is invaluable when fighting performance problems.</p>
