## Answers
### Answer ID: 56150040
<p>You can also disable the nested loop joins using <code>SET enable_nestloop = off</code> before running the query (IDK if that's possible to include with prisma).</p>

<p>That gives me an even slightly better execution time as with <code>ORDER BY ... + 0 DESC</code> without modifying the query at all: <a href="https://explain.depesz.com/s/Mi4W" rel="nofollow noreferrer">https://explain.depesz.com/s/Mi4W</a></p>

<p>I've created a full online example with slightly reduced data set (but still takes a while to load) to test the different queries: <a href="https://dbfiddle.uk/?rdbms=postgres_10&amp;fiddle=2c4d104804f57e1a59f7ed31bd57e2f5" rel="nofollow noreferrer">https://dbfiddle.uk/?rdbms=postgres_10&amp;fiddle=2c4d104804f57e1a59f7ed31bd57e2f5</a></p>

<p>If you're working with prisma, it probably would also be a good idea to share your prisma request from the client that's resulting into this SQL query. Maybe something can be done on that side as well.</p>

<p>From the query perspective, probably an optimization fence using a CTE gives the best possible result:</p>

<pre class="lang-sql prettyprint-override"><code>WITH
    cte_reviews (id) AS (
        SELECT r.id
        FROM review AS r
            INNER JOIN category_to_product AS cp ON (r.product = cp.product)
        WHERE
            cp.category IN (
                SELECT 27
                UNION ALL
                SELECT id FROM category WHERE parent = 27
            )
        ORDER BY 1 ASC
    )
SELECT id
FROM cte_reviews
ORDER BY id DESC
LIMIT 11 OFFSET 0;
</code></pre>

<p>So here, we're enforcing a forward index (only) scan and then just reverse and limit the result of that, which is much faster in this special case.</p>

Execution time: 22.021 ms
</code></pre>

### Answer ID: 56126376
<p>My guess is that due to an adversary distribution of the rows in the tables, the interesting rows come up last when PostgreSQL tries to use index scans to get the correct order.</p>

<p>Try to avoid index scans and use an explicit sort by changing the <code>ORDER BY</code> clause to</p>

<pre><code>ORDER BY "Alias".id + 0 DESC
</code></pre>

<p>What do I mean by &ldquo;adversary distribution&rdquo;? Based on its estimates, PostgreSQL thinks that there are quite a lot of rows that satisfy the condition, so it thinks that it is cheapest if it processes the rows in descending <code>Alias.id</code> order and just keeps going until it has found 11 rows satisfying the condition. Even if the guess is right, it may be that the (many) rows that satisfy the condition all have low <code>Alias.id</code>, so it has to compute way more rows than it bargained for.</p>

<p>Seeing your second execution plan, I suspect that at least part of the problem is that PostgreSQL over-estimates the number of rows that satisfy the condition: 1490226 instead of 13589 rows. Simplifying the query might help.</p>
