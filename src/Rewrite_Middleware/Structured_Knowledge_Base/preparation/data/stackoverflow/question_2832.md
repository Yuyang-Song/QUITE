# Prevent SQL Server 2017 from executing non correlated subquery in loop
[Link to question](https://stackoverflow.com/questions/54674980/prevent-sql-server-2017-from-executing-non-correlated-subquery-in-loop)
**Creation Date:** 1550075149
**Score:** 2
**Tags:** sql-server, performance
## Question Body
<p>We have a multi-tenant database, data separation is achieved with a key attached to all tables. The number of rows in each table may be very different for each tenant (0 to 500000+).
Since migrating to SQL Server 2017 sometimes very bad execution plans are used.
We didn't have any problems with SQL Server 2008 previously used.</p>

<p>The biggest problem is, that sometimes non-correlated sub queries are executed in a loop for each row of the outer query. The subquery would take 0 seconds, and the outer query using an IN(list of ids) also takes 0 seconds, but together they run for several minutes.</p>

<p>I have tried different solutions:</p>

<ul>
<li><code>OPTION(RECOMPILE)</code></li>
<li>Changing to a <code>JOIN</code> and <code>OPTION(FORCE ORDER)</code></li>
<li><code>UPDATE STATISTICS</code> for the tables</li>
<li>Parameter Sniffing is enabled but doesn't make a difference</li>
</ul>

<p>none of these solved the problem.</p>

<p>Example Query:</p>

<pre><code>SELECT address.adrId, address.adrCity FROM address 
WHERE address.orgId=1 AND
address.adrId IN (SELECT instAddress.adrId 
                    FROM instAddress
                    WHERE instAddress.instId = 12345 
                    AND instAddress.orgId = 1) 
</code></pre>

<p>UPDATE: actual execution plan, number of executions and estimated number of executions highlighted.
<a href="https://i.sstatic.net/dxVXD.png" rel="nofollow noreferrer"><img src="https://i.sstatic.net/dxVXD.png" alt="enter image description here"></a></p>

<p>Is it somehow possible to tell SQL Server to always execute the sub query first and only once? </p>

<p>This one query changed to the correct execution plan after latest update of statistics. This one and similar queries are generated by our application dynamically. We would like to prevent us from rewriting the complete database tier.</p>

## Answers
### Answer ID: 54675231
<p>I think that rewrite of the original query to use <code>EXISTS</code> can lead to more robust query plans:   </p>

<pre><code>SELECT a.adrId, a.adrCity FROM address  a
WHERE EXISTS ( SELECT *
                    FROM instAddress i
                    WHERE i.instId = 12345 
                    and i.orgId=1
                    AND i.orgId = d.orgId 
                    AND i.adrId = a.adrId 

             )
</code></pre>

<p><strong>Another observation</strong> is that such index will increase a chance that query optimizer will set a correlated query as an initial one to execute and then  fetch rows from [address]</p>

<pre><code>CREATE INDEX IX_1 ON instAddress (instId) INCLUDE (orgid, adrId )
</code></pre>

<p><strong>Another possible reason</strong> - outdated statistics.</p>

<p>According to OP:</p>

<blockquote>
  <p>The query in the initial example was one of about 10 similar queries
  for which this issue occurred, some of them appear and disappear when
  the statistics or the parameters change, but eventually pop up gain
  after a while</p>
</blockquote>

<p>If compatibility level is lower than <strong>130</strong>, default threshold of statistics to be auto updated is 20% changed rows. This is especially not nice for large tables and can be fixed by:</p>

<pre><code>DBCC TRACEON (2371, -1)
</code></pre>

<p><strong>Another possible reason</strong>: New cardinality estimator and correlated queries. </p>

<blockquote>
  <p>Since migrating to SQL Server 2017 sometimes very bad execution plans
  are used. We didn't have any problems with SQL Server 2008 previously
  used.</p>
</blockquote>

<p>The old (legacy) cardinality estimation (that was available in SQL 2008) can be enabled as a query hint, on a database level or on instance level, via trace flag</p>

<pre><code>SELECT .. FROM .. WHERE ..
OPTION (USE HINT ('FORCE_LEGACY_CARDINALITY_ESTIMATION'));  
</code></pre>

<p>or on a database level:</p>

<pre><code>USE DB
ALTER DATABASE SCOPED CONFIGURATION SET LEGACY_CARDINALITY_ESTIMATION = ON
</code></pre>

### Answer ID: 54675426
<p>You can also achieve this with a standard join:</p>

<pre><code>SELECT a.adrId, a.adrCity 
FROM address  a
inner join instAddress i on i.orgId = a.ordId and i.adrId = a.adrId
where i.instId = 12345 and a.orgId = 1
</code></pre>
